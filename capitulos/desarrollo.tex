\section{Levantamiento de requerimientos del área de Finanzas}

Se realizan reuniones con el área de Finanzas con el objetivo de identificar sus necesidades de información, incluyendo los reportes requeridos, los indicadores clave y la periodicidad de actualización de los mismos. Como parte de este proceso, se analiza el procedimiento actual de extracción manual de datos desde el sistema ERP, así como el formato utilizado por el departamento financiero para la consolidación de la información.

\subsection{Requerimientos de Información Financiera y Contable}

Se establecen los siguientes requerimientos para la solución tecnológica:

\begin{enumerate}
    \item \textbf{Automatización Integral de la Cédula de Gastos:} Se requiere la automatización completa de la cédula de gastos para eliminar la dependencia operativa de archivos de Excel complejos.
    \item \textbf{Consolidación Transaccional (Odoo y COI) y Modelado Dimensional:} Es imperativo consolidar la información de egresos proveniente de múltiples orígenes, unificando los datos del ERP principal (Odoo) con los sistemas contables adicionales.
    \item \textbf{Desarrollo de Vistas de Detalle para Validación de Integridad:} Se debe habilitar una vista analítica a nivel de transacción detallada que permita auditar la asignación de gastos.
    \item \textbf{Desarrollo de Vistas Agregadas para Conciliación de Saldos:} Se requiere una vista con la información agregada a nivel de cuenta de gasto para permitir una confrontación y cuadre directo contra la balanza de comprobación.
\end{enumerate}

\section{Diseño de arquitectura del sistema}

Se diseña la arquitectura tecnológica del sistema con el objetivo de establecer una solución que permita la extracción, procesamiento, almacenamiento y visualización automatizada de los datos financieros. Se define una infraestructura cloud en AWS como entorno principal de ejecución.

\subsection{Diseño de arquitectura Data Lakehouse}

El diseño se estructura bajo un modelo de Data Lakehouse fundamentado en tres pilares: robustez, flexibilidad y confiabilidad. Se utilizan los siguientes componentes:

\begin{itemize}
    \item \textbf{Cómputo Elástico (Amazon EC2):} Se utilizan instancias para el procesamiento de datos.
    \item \textbf{Almacenamiento Híbrido (EBS y S3):} Se provisionan volúmenes EBS para el Data Warehouse y Amazon S3 para el Data Lake.
    \item \textbf{Seguridad y Gestión de Accesos (IAM):} Se asignan Roles IAM y Grupos de Seguridad para el control del tráfico y permisos.
\end{itemize}

\section{Creación de scripts ETL de datos con Airflow}

Se desarrollan scripts en Python para implementar los procesos de extracción, transformación y carga de datos. Se utiliza Apache Airflow para definir flujos de ejecución mediante DAGs (\textit{Directed Acyclic Graphs}).

\subsection{Scripts de Extracción a Bronce}

La Capa Bronce constituye la primera capa del Data Warehouse. Su objetivo consiste en almacenar una copia de los datos lo más cercana posible a la fuente original, sin aplicar transformaciones de negocio significativas. El flujo general de ingesta sigue tres etapas: extracción SQL a CSV, conversión de CSV a Parquet mediante DuckDB y carga masiva desde Parquet hacia el Data Warehouse en PostgreSQL.

A continuación se describe el proceso utilizando como ejemplo la ingesta de la tabla \texttt{account\_move}, que corresponde a los movimientos contables y facturas del sistema ERP Odoo.

\subsubsection{Inicio de la instancia de ingesta en Airflow}

La ingesta se modela como un DAG de Apache Airflow. Para la tabla \texttt{account\_move}, el DAG se identifica como \texttt{bronce\_account\_move\_etl\_v19-2}. Se configura sin calendario automático (\texttt{schedule=None}), con \texttt{catchup=False} y \texttt{max\_active\_runs=1}, lo que indica una ejecución controlada que evita múltiples corridas simultáneas del mismo DAG.

Dentro de la configuración, se generan ventanas mensuales que abarcan desde el año 2022 hasta marzo de 2026. Cada ventana contiene una fecha inicial y una fecha final, como se muestra a continuación:

\begin{lstlisting}[language=Python, breaklines=true]
{"start": "2025-01-01", "end": "2025-01-31"}
\end{lstlisting}

Esta estrategia de ventanas permite dividir la extracción en lotes manejables, evitando problemas de memoria o cargas excesivamente grandes.

\subsubsection{Definición de la ventana de extracción}

Cada instancia de ingesta recibe una ventana con los parámetros \texttt{start\_date} y \texttt{end\_date}. A partir de estos valores, se construyen las rutas temporales para almacenar los archivos intermedios del mes procesado:

\begin{lstlisting}[language=Python, breaklines=true]
start_date = window["start"]
end_date   = window["end"]
csv_path     = "/tmp/account_move_YYYY_MM.csv"
parquet_path = "/tmp/account_move_YYYY_MM.parquet"
\end{lstlisting}

Esta lógica se ejecuta dentro de la tarea \texttt{extract\_to\_parquet}.

\subsubsection{Lectura del SQL de extracción}

La consulta de extracción se encuentra almacenada en el directorio \texttt{sql/bronce/account\_move.sql}. El DAG lee este archivo como una plantilla SQL y reemplaza los parámetros \texttt{\{\{ start\_date \}\}} y \texttt{\{\{ end\_date \}\}} por las fechas reales de la ventana correspondiente.

La consulta filtra los registros por rango de fechas, procesando únicamente los movimientos del periodo asignado:

\begin{lstlisting}[language=SQL, breaklines=true]
FROM account_move
WHERE "date" >= '{{ start_date }}'::date
  AND "date" <  '{{ end_date }}'::date
ORDER BY "date" ASC, id ASC;
\end{lstlisting}

\subsubsection{Conexión a la base de datos fuente}

La conexión hacia la base de datos de producción de Odoo se obtiene desde Apache Airflow mediante la función:

\begin{lstlisting}[language=Python, breaklines=true]
BaseHook.get_connection("potranca19_prod")
\end{lstlisting}

Dicha conexión contiene los parámetros de \textit{host}, puerto, usuario, contraseña y nombre de la base de datos del origen PostgreSQL.

\subsubsection{Extracción de datos a CSV}

La extracción se realiza mediante el comando \texttt{psql} utilizando la directiva \texttt{\textbackslash copy}, la cual envía el resultado de la consulta directamente hacia un archivo CSV local:

\begin{lstlisting}[language=bash, breaklines=true]
\copy ({query}) TO STDOUT WITH CSV HEADER
\end{lstlisting}

El DAG escribe el resultado directamente en disco (\texttt{stdout=csv\_file}), lo cual resulta fundamental porque evita cargar la totalidad del resultado en memoria. Para tablas de gran volumen como \texttt{account\_move}, este enfoque de transmisión directa (\textit{streaming}) reduce significativamente el riesgo de saturar la memoria RAM.

\subsubsection{Validación de la extracción}

Tras generar el archivo CSV, el DAG verifica si este contiene registros. Se calcula el número de filas restando la línea de encabezado:

\begin{lstlisting}[language=Python, breaklines=true]
row_count = sum(1 for _ in f) - 1
\end{lstlisting}

Si el mes no contiene datos, se elimina el archivo CSV y se devuelve un \textit{payload} con \texttt{parquet\_path: None}, indicando a la tarea siguiente que no debe realizar ninguna carga para esa ventana.

\subsubsection{Conversión de CSV a Parquet con DuckDB}

Cuando el CSV contiene registros válidos, DuckDB lo convierte al formato Parquet:

\begin{lstlisting}[language=SQL, breaklines=true]
COPY (
    SELECT *
    FROM read_csv_auto(...)
)
TO '{parquet_path}'
(FORMAT PARQUET, COMPRESSION 'ZSTD', ROW_GROUP_SIZE 100000);
\end{lstlisting}

En esta ingesta, el DAG fuerza todos los campos a \texttt{VARCHAR} para evitar conflictos de tipos durante la carga histórica. La resolución de tipos reales se delega a las capas posteriores, principalmente la Capa Plata. Esta decisión refuerza un principio clave de la Capa Bronce: no limpiar ni interpretar el dato en esta etapa, priorizando su recepción y conservación íntegra.

Una vez completada la conversión, se elimina el archivo CSV temporal para reducir el uso de almacenamiento:

\begin{lstlisting}[language=Python, breaklines=true]
if os.path.exists(csv_path):
    os.remove(csv_path)
\end{lstlisting}

La tarea devuelve un diccionario con las fechas de la ventana y la ruta del archivo Parquet generado, el cual se transfiere como \textit{payload} a la tarea de carga.

\subsubsection{Preparación de la carga hacia el Data Warehouse}

La siguiente tarea, denominada \texttt{load\_parquet\_to\_dw}, recibe el \textit{payload} anterior. Si el valor de \texttt{parquet\_path} es \texttt{None}, la tarea concluye sin ejecutar la carga.

En caso contrario, se obtiene la conexión al Data Warehouse mediante:

\begin{lstlisting}[language=Python, breaklines=true]
BaseHook.get_connection("potranca19_DLH")
\end{lstlisting}

Con los parámetros obtenidos se construye un DSN (\textit{Data Source Name}) de PostgreSQL.

\subsubsection{Conexión de DuckDB al Data Warehouse}

Los scripts de carga utilizan la extensión \texttt{postgres} de DuckDB para conectarse directamente al Data Warehouse:

\begin{lstlisting}[language=SQL, breaklines=true]
INSTALL postgres;
LOAD postgres;
ATTACH '{{ pg_dsn }}'
AS pg_dw (TYPE POSTGRES);
\end{lstlisting}

Esta configuración permite que DuckDB lea el archivo Parquet local y escriba directamente en PostgreSQL sin intermediarios adicionales.

\subsubsection{Creación del esquema y tabla destino}

Antes de ejecutar la carga, el script crea el esquema Bronce si este no existe:

\begin{lstlisting}[language=SQL, breaklines=true]
CREATE SCHEMA IF NOT EXISTS pg_dw.bronce;
\end{lstlisting}

Asimismo, se crea la tabla destino utilizando la estructura del archivo Parquet, sin insertar filas inicialmente:

\begin{lstlisting}[language=SQL, breaklines=true]
CREATE TABLE IF NOT EXISTS pg_dw.bronce.account_move AS
SELECT * FROM read_parquet('{{ parquet_path }}')
WHERE 1=0;
\end{lstlisting}

\subsubsection{Aplicación de idempotencia y carga de datos}

Para garantizar que la ingesta pueda repetirse sin generar duplicados, se aplica una lógica de refresco que elimina previamente los registros del rango a recargar:

\begin{lstlisting}[language=SQL, breaklines=true]
DELETE FROM pg_dw.bronce.account_move
WHERE "date" >= '{{ start_date }}'
  AND "date" <  '{{ end_date }}';
\end{lstlisting}

Este paso resulta clave: si una instancia falla o se requiere reprocesar un mes, se eliminan primero los registros del rango correspondiente y luego se vuelven a insertar.

Finalmente, DuckDB carga el contenido del Parquet hacia PostgreSQL:

\begin{lstlisting}[language=SQL, breaklines=true]
INSERT INTO pg_dw.bronce.account_move
SELECT * FROM read_parquet('{{ parquet_path }}');
\end{lstlisting}

\subsubsection{Limpieza de archivos temporales}

Tras ejecutar la carga, el DAG cierra la conexión de DuckDB y elimina el archivo Parquet temporal:

\begin{lstlisting}[language=Python, breaklines=true]
if os.path.exists(parquet_path):
    os.remove(parquet_path)
\end{lstlisting}

Esto deja la instancia en un estado limpio y evita la acumulación de archivos en el directorio temporal.


\subsection{Ingesta de datos del Modelo de datos Bronce a Plata con SQLmesh}

A diferencia de la Capa Bronce, donde se extrae información directamente desde el sistema origen, la Capa Plata se construye a partir de los datos ya existentes en las tablas \texttt{bronce.*} dentro de PostgreSQL. Para esta transformación se utiliza SQLMesh, un marco de trabajo que permite definir modelos SQL declarativos con control de versiones y materialización automatizada.

El flujo general se resume de la siguiente manera: SQLMesh lee las tablas crudas del esquema \texttt{bronce}, aplica transformaciones de estandarización y materializa los resultados en el esquema \texttt{plata}.

\subsubsection{Ejecución del plan de SQLMesh desde Airflow}

El DAG encargado de iniciar el proceso se denomina \texttt{sqlmesh\_plan}. Dentro de este, un \texttt{BashOperator} activa el entorno virtual de Python, se posiciona en el directorio \texttt{sql/} y ejecuta el siguiente comando:

\begin{lstlisting}[language=bash, breaklines=true]
sqlmesh plan prod --auto-apply
\end{lstlisting}

Este comando indica a SQLMesh que debe calcular el plan de cambios pendientes y aplicarlo automáticamente en el ambiente de producción (\texttt{prod}). El DAG inyecta las credenciales del Data Warehouse mediante variables de entorno (\texttt{PG\_HOST}, \texttt{PG\_USER}, \texttt{PG\_PASSWORD}, \texttt{PG\_DATABASE} y \texttt{PG\_PORT}), las cuales se obtienen de la conexión de Airflow \texttt{potranca19\_DLH}.

\subsubsection{Configuración de conexión de SQLMesh}

La configuración de SQLMesh se encuentra en el archivo \texttt{sql/config.py}, donde se define el \textit{gateway} principal denominado \texttt{potranca19\_DLH} y la conexión PostgreSQL que utiliza las variables de entorno previamente inyectadas. También se establece que el dialecto por defecto de los modelos es PostgreSQL, lo que significa que todas las transformaciones se compilan y ejecutan como sentencias SQL nativas del motor del Data Warehouse.

\subsubsection{Descubrimiento y estructura de los modelos Plata}

Los modelos de la Capa Plata se ubican en el directorio \texttt{sql/models/plata/} y siguen la convención de nomenclatura \texttt{plata\_\_<entidad>.sql}. Cada archivo inicia con un bloque \texttt{MODEL} que declara las propiedades del modelo. Por ejemplo, para la tabla \texttt{account\_move}:

\begin{lstlisting}[language=SQL, breaklines=true]
MODEL (
    name plata.account_move,
    kind FULL,
    grain (id),
    description 'Carga bronce.account_move -> plata.account_move 
                 con casteos y valores NOT NULL'
);
\end{lstlisting}

La propiedad \texttt{kind FULL} indica que el modelo se reconstruye por completo en cada ejecución, generando una regeneración integral de la tabla Plata con base en el estado actual de Bronce. La propiedad \texttt{grain} define la granularidad esperada del modelo, es decir, un registro por identificador.

\subsubsection{Transformaciones aplicadas en la Capa Plata}

Después del bloque \texttt{MODEL}, cada archivo contiene la sentencia \texttt{SELECT} que define la transformación. La fuente de datos siempre es una tabla del esquema \texttt{bronce}. Las transformaciones que se aplican de manera general incluyen:

\begin{enumerate}
    \item \textbf{Casteo de tipos:} Cada columna se convierte explícitamente al tipo de dato correcto. Los campos numéricos se transforman a \texttt{numeric} o \texttt{int4}, las fechas a \texttt{date} y los textos a \texttt{varchar}:
    \begin{lstlisting}[language=SQL, breaklines=true]
CAST(b.id AS int4) AS id,
b.amount_total::numeric AS amount_total,
b.date::date AS "date",
    \end{lstlisting}

    \item \textbf{Protección de campos obligatorios:} Se utiliza la función \texttt{COALESCE} para asignar valores predeterminados a campos críticos que no deben contener nulos:
    \begin{lstlisting}[language=SQL, breaklines=true]
COALESCE(b.date::date, CURRENT_DATE) AS "date",
COALESCE(b.state::varchar, 'draft') AS state,
COALESCE(b.journal_id::int4, 0) AS journal_id,
    \end{lstlisting}

    \item \textbf{Conversión de estructuras JSONB:} Los campos que provienen de Odoo como estructuras JSON se convierten explícitamente al tipo \texttt{jsonb} de PostgreSQL, permitiendo que las capas posteriores los consulten de forma nativa:
    \begin{lstlisting}[language=SQL, breaklines=true]
b.sending_data::jsonb AS sending_data,
b.analytic_distribution::jsonb AS analytic_distribution,
    \end{lstlisting}

    \item \textbf{Reglas de integridad de negocio:} Se implementan validaciones específicas según la entidad. Por ejemplo, en la tabla \texttt{account\_move\_line}, se evita la asignación de una cuenta contable a líneas que representan secciones o notas:
    \begin{lstlisting}[language=SQL, breaklines=true]
CASE
    WHEN b.display_type IN ('line_section','line_note')
    THEN NULL
    ELSE b.account_id::int4
END AS account_id
    \end{lstlisting}

    Asimismo, se protege la consistencia entre débito y crédito para evitar que una misma línea contenga valores simultáneos en ambos campos:
    \begin{lstlisting}[language=SQL, breaklines=true]
CASE WHEN b.credit::numeric > 0 THEN 0 
     ELSE b.debit::numeric END AS debit,
CASE WHEN b.debit::numeric > 0 THEN 0 
     ELSE b.credit::numeric END AS credit,
    \end{lstlisting}
\end{enumerate}

\subsubsection{Materialización y disponibilidad para la Capa Oro}

Al ejecutar \texttt{sqlmesh plan prod -{}-auto-apply}, SQLMesh determina qué modelos requieren actualización, ejecuta las transformaciones directamente dentro del motor PostgreSQL del Data Warehouse y materializa los resultados en el esquema \texttt{plata}. Conceptualmente, para cada modelo de tipo \texttt{FULL}, el proceso equivale a reconstruir la tabla completa a partir del \texttt{SELECT} definido.

Una vez completada la materialización, las tablas del esquema \texttt{plata} (como \texttt{plata.account\_move}, \texttt{plata.account\_move\_line}, \texttt{plata.account\_account} y \texttt{plata.res\_partner}) quedan tipadas, limpias y normalizadas, listas para ser consumidas por los modelos de la Capa Oro, donde se construyen las tablas de hechos y dimensiones del modelo dimensional.
